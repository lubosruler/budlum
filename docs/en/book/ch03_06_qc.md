# Chapter 3.6: Optimistic QC and PQ Attestation

Optimistic QC lets Budlum keep normal operation fast while still preparing for post-quantum verification.

## 1. Why "Optimistic"?

The chain commits compact roots and verifies heavy post-quantum material only when needed. Honest operation remains light; suspicious data can still be challenged.

## 2. Data Structure: `QcBlob`

`QcBlob` groups validator attestations, post-quantum signatures, Merkle roots, and metadata for a checkpoint. The block can commit to the blob without carrying every large signature directly in the header.

Each PQ signature is produced over `epoch`, `checkpoint_height`, `checkpoint_hash`, and `validator_index`. Binding `checkpoint_height` into the signing message prevents a signature from being reused in a different checkpoint context that happens to share other fields.

QC blobs are accepted only after the checkpoint height is valid, the local block hash matches, the Merkle root matches the included signatures, and every validator address plus Dilithium public key matches the epoch validator snapshot.

## 3. QC Fault Proof

If a validator includes an invalid PQ signature:

1.  A challenger prepares the bad signature and its Merkle path.
2.  A `QcFaultProof { proof_data }` message can be broadcast over P2P.
3.  The recipient parses the proof, verifies it against the stored `QcBlob` and epoch snapshot, then applies the verdict.

The current `InvalidDilithiumV1` proof invalidates finality/QC metadata but does not slash the validator. Future signed or ZK-backed proof kinds can return slashable verdicts.

## 4. Advantages

1.  **Performance:** heavy post-quantum math does not burden the main path.
2.  **Light blocks:** large signatures do not bloat headers.
3.  **Mandatory consistency:** finality certificate acceptance requires a verified QC blob.
4.  **Future safety:** Budlum can later move from optimistic mode to strict or ZK-backed verification.

## Summary

Optimistic QC balances today's performance with tomorrow's cryptographic requirements.
