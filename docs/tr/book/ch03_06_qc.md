# Bölüm 3.6: Optimistic QC ve PQ Attestation

Bu bölüm, Budlum projesinin kuantum bilgisayarlara karşı hazırlığı olan **QC sidecar** ve **Post-Quantum (PQ) Attestation** mekanizmasını detaylandırır.

Kaynak Dosya: `src/consensus/qc.rs`

---

## 1. Kavram: Neden QC Sidecar?

Kuantum sonrası kriptografi (Dilithium vb.), klasik Ed25519 imzalarından çok daha ağır ve büyüktür. Eğer her blokta 100 kişinin PQ imzasını doğrudan ana zincirde doğrulasaydık, ağ performansı ciddi oranda düşerdi.

Budlum bu sorunu sidecar yaklaşımıyla çözer:
- PQ imzaları ana blok başlığına değil, yan kanal olan `QcBlob` içine gömülür.
- `FinalityCert` geldiğinde node, o checkpoint için `QcBlob` yoksa ağdan ister.
- Gelen blob, Merkle root + Dilithium signature + validator identity açısından doğrulanmadan kabul edilmez.

Yani akış hâlâ “blok gövdesine her imzayı koymayan” hafif bir modeldir; ama doğrulama artık yalnızca şikayet geldiğinde yapılan gevşek bir kontrol değildir. Finality kabulü, doğrulanmış QC blob'a bağlanmıştır.

---

## 2. Veri Yapısı: `QcBlob`

`QcBlob`, bir checkpoint için toplanan tüm PQ imzalarını barındıran veri paketidir.

```rust
pub struct QcBlob {
    pub epoch: u64,
    pub checkpoint_height: u64,
    pub checkpoint_hash: String,
    pub pq_signatures: Vec<PqSignatureEntry>,
    pub merkle_root: String,
    pub created_epoch: u64,
}
```

`PqSignatureEntry` her validator için şu bilgiyi taşır:

```rust
pub struct PqSignatureEntry {
    pub validator_index: u32,
    pub validator_address: String,
    pub dilithium_signature: Vec<u8>,
}
```

- **Validator Identity Bağı:** `validator_index` ve `validator_address`, `ValidatorSetSnapshot` içindeki kayıtla eşleşmek zorundadır.
- **İmza Mesajı:** Her PQ imzası `pq_signing_message(epoch, checkpoint_height, checkpoint_hash, validator_index)` üzerinde üretilir. `checkpoint_height` imzaya dahil edilerek aynı hash/epoch kombinasyonunun farklı checkpoint bağlamlarında tekrar kullanılmasının önüne geçilir.
- **Depolama:** Doğrulanmış bloblar diskte `QC_BLOB:{Height}` anahtarı altında saklanır.
- **P2P Akışı:** `QcBlobResponse` alındığında node blob'u parse eder, Merkle root'u hesaplar, ardından Dilithium imzalarını validator snapshot'ına karşı doğrular ve ancak sonra persist eder.

## 3. QC Fault Proof ve ZK Hazırlığı

Eğer bir `QcBlob` geçersiz bir PQ imzası içerirse düğüm bunu tespit edebilir. Bu artık doğrudan slash anlamına gelmez; kanıt doğrulama sonucu açık bir verdict üretir.

1. **Kanıt Oluşturma:** `QcBlob::detect_fault_proofs`, geçersiz Dilithium imzası gördüğünde ilgili yaprağın Merkle yolunu çıkarır.
2. **Doğrulama:** `QcFaultProof::verify_against_blob`, kanıtın blob'a dahil olduğunu ve gerçekten geçersiz imzaya işaret ettiğini kontrol eder.
3. **Verdict:** Bugünkü `InvalidDilithiumV1` kanıtı finality/QC invalidation üretir; validator slash işlemi yalnızca ileride daha güçlü signed veya ZK-backed kanıtlar için ayrı bir verdict olarak açılır.
4. **P2P Yayılımı:** `QcFaultProof { proof_data }` mesajı proof'u ağda taşır. Alıcı node proof'u JSON olarak parse eder, `ChainActor` üzerinden doğrulatır ve geçerliyse verdict'i uygular.

```rust
pub struct QcFaultProof {
    pub version: u8,
    pub epoch: u64,
    pub checkpoint_height: u64,
    pub checkpoint_hash: String,
    pub validator_index: u32,
    pub validator_address: String,
    pub kind: QcProofKind,
}

pub enum QcProofKind {
    InvalidDilithiumV1 { /* Merkle leaf + path */ },
    ZkInvalidAttestationV1 { /* proof bytes + public inputs */ },
}
```

Not: Fault proof işleyicisi (`Blockchain::handle_qc_fault_proof`) ve P2P taşıma mesajı birlikte mevcuttur. Slash kararı hâlâ verdict'e bağlıdır; mevcut invalid-Dilithium Merkle kanıtları finality invalidation üretir ama validator'ı doğrudan slash etmez.

---

## 4. Mekanizmanın Avantajları

1. **Performans:** PQ imzaları blok başlığına taşınmadığı için ana zincir gövdesi hafif kalır.
2. **Hafif Bloklar:** MB'larca tutan PQ imzaları blok başlığını şişirmez.
3. **Zorunlu Tutarlılık:** Finality cert kabulü artık doğrulanmış `QcBlob` gerektirir; sidecar tamamen opsiyonel bir ek olmaktan çıkmıştır.
4. **Verdict Mekanizması:** Hatalı PQ attestasyonları sadece “loglanan” anormallikler değildir; bugünkü kanıtlar finality invalidation üretir, ileride ZK veya daha güçlü imzalı kanıtlar slash verdict'i de üretebilir.

---

## Özet

QC sidecar mekanizması, Budlum'un uzun vadeli güvenlik stratejisinin PQ ayağını temsil eder:
- **Dilithium:** Modern PQ standardı.
- **Merkle Trees:** Devasa imzaların düzenli yönetimi.
- **Fault Proofs:** Geçersiz imzaları finality invalidation ile işleyen, ZK-backed ve slashable kanıtlara genişleyebilen denetim mekanizması.
